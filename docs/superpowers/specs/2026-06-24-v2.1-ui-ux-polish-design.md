# Elite Scope App v2.1 — UI/UX Polish — Design

Date: 2026-06-24
Status: Approved for planning

## Context

v2's AI-layer rebuild (see `docs/superpowers/specs/2026-06-23-v2-ai-layer-design.md` and
its implementation plan) shipped and was tested in real use on a desktop browser.
This round addresses concrete UI/UX feedback gathered from that testing — including
feedback captured through the app's own new feedback-button feature (GitHub issue #3) —
rather than speculative polish. Three of the original v2 feedback items are addressed
here; one (transcription latency) is explicitly deferred.

## Scope decisions from discovery

- **Transcription latency is deferred, not built now.** The cheap fix (switching to
  `whisper-1`) was rejected: `whisper-1` is generally less accurate than
  `gpt-4o-transcribe` on accented/noisy speech, which is specifically why
  `gpt-4o-transcribe` was chosen in v2. Trading accuracy for an unconfirmed speed
  gain isn't worth it. If lag is still a problem after the rest of this round ships,
  the real fix is chunked transcription (rolling ~3-5 second pieces while still
  talking, so only the last couple seconds need processing after mic release) — but
  that's a future round, only if needed.
- **Proxy URL field is removed entirely**, not hidden behind an "advanced" toggle.
  It has never been needed (JobTread allows direct browser calls) and a future
  re-add is a small, contained fix if JobTread's policy ever changes.
- **Readability sizing applies everywhere** (phone and desktop), not just desktop,
  even though the desktop layout is what surfaced the complaint.

## 1. Feedback button redesign

**Problem:** The feedback button (a bare icon) wasn't obviously a hold-to-talk
control, and it auto-sent immediately on release with no chance to review, edit,
or add more before sending.

**Design:**
- Replace the icon-only button with an icon **and a short text label**
  (e.g. "💬 Feedback") so its purpose is clear without trial and error.
- Tapping it opens a modal, reusing the existing Settings-modal pattern/markup
  already in `index.html` (same backdrop, sheet, topbar-with-close-button
  structure) — titled "Send Feedback" with one line explaining its purpose
  ("Tell us about a problem or an idea").
- Inside the modal: a hold-to-talk mic button using the same interaction
  language as the existing scope-line and notes mics (same visual states:
  idle → listening → transcribing).
- The transcript lands in an editable `<textarea>` inside the modal — not
  auto-sent. The user can edit it directly, or hold the mic again to
  transcribe more text, appended to whatever's already there (same
  insert-at-cursor/append behavior as the existing notes mic).
- Two explicit buttons: **Send** (files the GitHub issue via the existing
  `buildFeedbackIssue`/`fileGithubIssue` pipeline — unchanged) and **Cancel**
  (closes the modal, discards the draft, no issue filed).
- On successful send: existing toast ("Feedback sent — thanks!"), modal closes.

**What stays the same:** the transcription pipeline (`transcribeAudio`), the
GitHub-issue-filing pipeline (`buildFeedbackIssue`, `fileGithubIssue`,
`functions/github-issue.js`) — only the front-end flow around capture changes,
from "auto-send on release" to "review, edit, then explicit send."

## 2. Settings & Setup cleanup

**Problem:** The Proxy URL field clutters both the initial Setup screen and the
Settings modal, and has never been needed.

**Design:** Remove the Proxy URL field, its label, and its hint text from both
screens. Remove the `proxyUrl` global variable and its two `localStorage`
read/write sites (`ec_proxy` key) in `init()`, `saveSetup()`, `openSettings()`,
`saveSettings()`. Remove its use in `jtPost()` (the function that currently
checks for a proxy URL and routes through it if set) — `jtPost()` always calls
JobTread directly going forward. Also remove the stale "CORS NOTE" wording in
the file's header comment block that still references the proxy as a fallback
option, replacing it with a one-line note that JobTread is called directly.

## 3. Readability & desktop layout

**Problem:** Text and controls feel small and hard to read/hit for a user in
his mid-50s; on a normal desktop monitor, the layout stretches edge-to-edge,
pushing controls (the line delete "✕", the drag handle) to the far edges of
the screen, requiring scrolling/reaching to use them. Drag-and-reorder doesn't
visually move the whole line as a unit, making it unclear where a line is
being dropped.

**Design:**
- **Sizing:** increase base font sizes and touch-target sizes (buttons, the
  mic buttons, line-item padding, input fields) by roughly 20% across the
  board, applied uniformly on phone and desktop — not a separate "desktop
  mode." This matches the user's own reference point (the app feels right at
  120% browser zoom on a normal desktop monitor).
- **Desktop layout:** add a centered max-width container (applied at the
  top-level layout, e.g. around `.card` elements and the header) so the app
  doesn't stretch full-width on wide screens. Controls stay within a
  comfortable, consistent reading/reaching width regardless of monitor size,
  instead of spreading to the edges. On narrow (phone) viewports this has no
  visible effect — the content already fills the available width there.
- **Drag-and-reorder visual feedback:** the dragged `<li class="line-item">`
  already gets a `.dragging` class with reduced opacity (existing CSS). The
  fix is ensuring the *entire* line — drag handle, line number, text, delete
  button, all wrapped in the `.line-row` div from the v2 flagging work —
  visually moves together as one block during drag, rather than any
  per-element drift. This is primarily a verification/cleanup of the existing
  `.line-row` wrapper's drag styling, not a new structural change.

## Explicitly out of scope for this round

- Transcription latency fix (chunked transcription) — deferred, revisit only
  if still a problem after this round ships and gets used.
- Any change to the JobTread integration, AI review/flagging logic,
  transcription pipeline internals, or the GitHub-issue-filing backend —
  all unchanged; this round is front-end/UX only.

## Open question to verify in real use

Does removing the Proxy URL field ever cause a problem if JobTread's CORS
policy changes in the future? Not expected, but worth knowing this is the
trade-off if "Failed to fetch" errors on the job-loading screen ever
reappear — the fix at that point is re-adding a small proxy function
(the same pattern already used for `/transcribe` and `/github-issue`),
not reintroducing the old unused field.
